# Iris — AI Features Brainstorm

- **Date**: 2026-03-01
- **Status**: Brainstorm complete

---

## Overview

This document catalogs every AI-powered feature under consideration for Iris, organized by functional category. Each feature includes what it does, how it is triggered, what AI capability it requires, and a concrete example interaction. Features range from near-term essentials to blue-sky experiments.

## 1. Inbox Intelligence

Features that automatically analyze, score, and organize incoming email without user intervention.

| # | Feature | What It Does | User Trigger | AI Requirement | Example Interaction |
|---|---------|-------------|--------------|----------------|-------------------|
| 1.1 | **Smart Categorization** | Goes beyond Gmail's 5 tabs. Learns the user's mental model over time and creates dynamic categories (e.g., "Shipping Updates", "Receipts", "GitHub Notifications") that reflect how they actually think about email. | Automatic | Classification model (1-3B parameters) | Iris creates a custom "Client Escalations" bucket after detecting frustrated tone from known clients across several threads. The user never configured this category — Iris inferred it from behavior. |
| 1.2 | **Priority Scoring** | Applies an Eisenhower matrix (urgent + important) to every email. Factors include sender relationship strength, content intent, detected deadlines, thread activity/heat, and historical engagement patterns. | Automatic on every incoming email | Classification model + LLM (3-7B) | An email from the user's boss about Q3 budget due Friday scores as urgent+important and surfaces at the top. A marketing newsletter scores as not urgent+not important and drops to the bottom. |
| 1.3 | **Relationship-Aware Prioritization** | Weights priority by relationship strength rather than treating all senders equally. Boss > random LinkedIn connection. Learns from interaction patterns — reply speed, thread depth, frequency. | Automatic; learns from interaction patterns | Graph analysis + classification model | An email from someone the user consistently replies to within 10 minutes gets higher priority than an email from someone the user typically takes 2 days to reply to, even if the content is similar. |
| 1.4 | **Intent Detection** | Classifies every email into an intent category: asking you something, informing you, selling you, requesting action, FYI, social, or transactional. This intent label drives downstream features like priority and "Needs Reply." | Automatic | LLM (3-7B) | "Can you review the attached?" is classified as ACTION_REQUEST. "FYI the server migration completed" is classified as INFORMATIONAL. "50% off all plans this week!" is classified as SALES. |
| 1.5 | **Deadline Extraction** | Detects dates and deadlines mentioned in email content. Creates tasks with deadlines linked to the originating thread. Handles relative dates ("by end of week"), explicit dates, and implied urgency. | Automatic | NER + LLM (3-7B) | "Please have this reviewed by March 15" triggers creation of a task with a March 15 deadline, linked to the thread. The task appears in the user's task view with the source email referenced. |
| 1.6 | **Sentiment Analysis** | Tracks emotional tone of incoming email. Surfaces frustrated, urgent, or happy signals as metadata on the email. Can be used to escalate or de-prioritize. | Automatic | Sentiment model (1-3B) | An email containing "I'm extremely disappointed with the response time" is flagged with negative sentiment. This feeds into priority scoring, pushing it higher in the queue. |
| 1.7 | **Thread Importance Decay** | Old threads lose priority over time unless reactivated by new activity. Prevents stale threads from cluttering priority views. Decay rate is configurable and adapts to the user's typical response cadence. | Automatic | Statistical model | A 2-week-old thread with no new activity gradually drops in priority. If someone replies to it, importance resets to reflect the new activity. |
| 1.8 | **"Needs Reply" Detection** | Identifies emails that expect the user's response and flags cases where no reply has been sent. Considers question marks, direct asks, CC vs TO position, and sender expectations. | Automatic | LLM (3-7B) | An email asking "Could you confirm?" has been sitting for 3 days with no reply. It surfaces in a dedicated "Needs Reply" queue with the age highlighted. |
| 1.9 | **Duplicate/Related Clustering** | Groups emails about the same topic from different senders into clusters. Reduces cognitive load when multiple people email about the same event or issue. | Automatic | Embedding model for semantic similarity | Three different people email about a server outage. Instead of appearing as 3 separate threads, they are clustered together with a shared topic label: "Server Outage — 3 threads." |
| 1.10 | **VIP Auto-Detection** | Learns who matters based on the user's behavior rather than requiring manual VIP setup. Considers reply speed, thread engagement depth, and communication frequency. | Automatic | Behavioral analysis | The user always replies to sarah@company.com within 15 minutes, engages in long threads, and initiates conversations. Sarah is automatically promoted to VIP status. |

## 2. Composing & Writing

Features that help users write emails faster, better, and in the right tone.

| # | Feature | What It Does | User Trigger | AI Requirement | Example Interaction |
|---|---------|-------------|--------------|----------------|-------------------|
| 2.1 | **Draft from Intent** | User describes what they want to say in plain English. AI generates a full email draft with appropriate structure, greeting, body, sign-off, and tone. | Manual — via chat interface or command palette | LLM (7B+) | User types: "tell Sarah the project is delayed 2 weeks, apologetic tone." AI generates a full draft with subject line, apology, explanation, revised timeline, and professional sign-off. |
| 2.2 | **Multi-Option Reply** | Generates 2-3 reply options with different tones or approaches (formal, casual, brief). User picks one and optionally edits. Reduces decision fatigue. | Manual — triggered when replying | LLM (7B+) | For a meeting request, Iris shows three options: (1) Formal: "Thank you for the invitation. I would be pleased to attend..." (2) Casual: "Sounds great, count me in!" (3) Brief: "Confirmed. See you then." |
| 2.3 | **Writing Style Learning** | Analyzes the user's sent email history to learn their voice — greetings, sign-offs, sentence structure, tone, vocabulary, formality level. Drafts then match the user's style. | Automatic after initial training period | LLM (7B+) fine-tuned or prompted with sent mail examples | After analyzing 200+ sent emails, AI drafts use the user's characteristic greeting ("Hey team,"), sign-off ("Best, D"), and sentence patterns. Output is indistinguishable from user-written email. |
| 2.4 | **Multi-Language Compose** | User writes in their preferred language; AI translates the email for the recipient. Preserves tone and intent across languages. | Manual — language toggle in compose | LLM (7B+) with translation capability | User composes in English. Toggles target to Japanese. AI translates the full email while preserving formality level appropriate for Japanese business communication. |
| 2.5 | **Context-Aware Autocomplete** | Knows the full thread history and suggests completions that reference prior discussion points, decisions, and people. Goes beyond generic autocomplete. | Automatic while typing | LLM (7B+) | User starts typing "As we discussed in..." and autocomplete suggests "...last Tuesday's sync, the API migration deadline is March 20" based on actual thread content. |
| 2.6 | **Formal/Informal Toggle** | Rewrites the same email content in a different register — toggle between formal and casual without changing the substance. | Manual — toggle button in compose | LLM (7B+) | Casual: "Hey, sounds good! Let's do it." toggles to Formal: "Thank you for your proposal. I agree with the suggested approach and would like to proceed." |
| 2.7 | **Smart CC/BCC Suggestions** | Suggests who should be on the email based on thread context, topic, and organizational patterns. Prevents "forgot to CC" situations. | Automatic suggestion (user accepts/rejects) | Graph analysis + LLM | User replies about budget allocation. | 2.8 | **Subject Line Generation** | Generates clear, descriptive subject lines based on email content. | 2.9 | **Template Auto-Generation** | Detects repeated email patterns — emails the user writes frequently with similar structure — and offers to create reusable templates. | Proactive suggestion | Pattern detection + LLM | "You've sent 5 similar onboarding welcome emails this month. Would you like to save this as a template?" User confirms. Template is created with variable fields (name, start date, team). |
| 2.10 | **Grammar & Tone Check** | Real-time writing quality check with contextual suggestions. Goes beyond grammar to catch tone issues, passive aggression, unclear asks, and overly long emails. | Automatic while composing | LLM (3-7B) | Flags "per my last email" as potentially passive-aggressive and suggests "as mentioned previously" or rewording entirely. Also flags that the key request is buried in paragraph 4. |

## 3. Search & Retrieval

Features that let users find information across their entire email history using natural language and semantic understanding.

| # | Feature | What It Does | User Trigger | AI Requirement | Example Interaction |
|---|---------|-------------|--------------|----------------|-------------------|
| 3.1 | **Natural Language Search** | Users ask questions in plain English instead of constructing keyword queries. AI interprets intent, searches, and returns relevant results with context. | Manual — search bar or chat | LLM (7B+) + embedding retrieval (RAG) | User asks: "what did the design team decide about the logo last month?" Iris returns the specific email thread where the decision was made, with the decision highlighted. |
| 3.2 | **Semantic Search** | Meaning-based search that goes beyond exact keyword matching. Understands synonyms, related concepts, and intent. | Manual — search bar | Embedding model (nomic-embed, all-MiniLM, or similar) | Searching "pricing discussion" also surfaces emails about "cost breakdown," "rate card," "fee structure," and "budget proposal" even though none contain the word "pricing." |
| 3.3 | **Cross-Thread Knowledge Graph** | Maps entities (people, projects, decisions, dates, amounts) across all threads into a queryable knowledge graph. Enables complex relationship queries. | Manual — via chat or dedicated view | LLM + graph construction | User asks "everything related to Project Falcon." Iris returns threads, key decisions, involved people, shared documents, and a timeline — assembled from 50+ threads spanning 6 months. |
| 3.4 | **Attachment Content Search** | Searches inside PDFs, Word documents, spreadsheets, and presentations attached to emails. Indexes attachment content alongside email body. | Manual — search bar | Document parsing + embedding model | Searching "revenue forecast" finds the term inside an attached Excel spreadsheet from 3 months ago, even though the email body only said "see attached." |
| 3.5 | **People-Centric Search** | Search by relationship combinations and contact attributes rather than just content. Combines people with topics for precise retrieval. | Manual — search bar or chat | Graph analysis + LLM | "Show me everything involving Sarah AND budget" returns all threads where Sarah participated that discuss budget-related topics, regardless of exact keywords used. |
| 3.6 | **Temporal Reasoning** | Understands time-relative queries and can map vague time references to actual dates based on context in the user's email history. | Manual — search bar or chat | LLM for time interpretation | User asks "emails from around when we launched v2." Iris knows v2 launched in October (from email context) and searches the October timeframe. |
| 3.7 | **Answer Extraction** | Instead of returning emails, directly answers the user's question with the answer extracted from email content. Cites the source email. | Manual — chat interface | LLM (7B+) + RAG | User asks: "what's the contract renewal date with Acme?" Iris responds: "April 15, 2026" and cites the source as "email from John on Feb 3, thread: Acme Contract Renewal." |

## 6. Security & Trust

Features that protect users from threats, detect anomalies, and build confidence in email authenticity.

| # | Feature | What It Does | User Trigger | AI Requirement | Example Interaction |
|---|---------|-------------|--------------|----------------|-------------------|
| 6.1 | **Contextual Phishing Detection** | Goes beyond pattern matching. Understands relationships, communication patterns, and context to detect sophisticated phishing attempts that rule-based systems miss. | Automatic on every incoming email | LLM (7B+) | "Your CEO would never ask you to buy gift cards via email. This sender matches the CEO's display name but uses a different domain. Flagged as phishing." |
| 6.2 | **Social Engineering Detection** | Detects psychological manipulation tactics: artificial urgency, authority exploitation, fear-based pressure, and unusual requests. | Automatic | LLM (7B+) | "This email creates artificial urgency ('must respond within 1 hour') and requests a wire transfer. This pattern is unusual for this sender, who normally sends weekly project updates." |
| 6.3 | **Impersonation Detection** | Catches display name spoofing, lookalike domains, and subtle address variations. Compares against known contact addresses. | Automatic | Address analysis + pattern matching | "This looks like Sarah but the address is s4rah@gmail.com instead of sarah@company.com. The display name matches your contact, but the address does not." |
| 6.4 | **Link Safety with Context** | Analyzes links not just for known threats but in the context of the email. A login link from an unknown domain in a "verify your account" email is suspicious even if the URL is new. | Automatic | URL analysis + LLM | "This link goes to a login page at acme-verify.com, but you've never interacted with this domain. The email asks you to 'verify your credentials.' Suspicious — do not click." |
| 6.5 | **Data Leak Prevention** | Warns before sending emails containing sensitive data: credit card numbers, SSNs, API keys, passwords, confidential markers. Scans outgoing email and attachments. | Automatic — before-send check | NER + classification model | "This email contains what appears to be a credit card number (ending in 4532). Are you sure you want to send this? Consider removing or encrypting the sensitive data." |
| 6.6 | **Anomaly Detection** | Detects unusual patterns in the user's account: emails sent at odd hours, new devices, bulk operations, or behavior that deviates from established patterns. | Automatic | Statistical analysis | "3 emails were sent at 3am from a new IP address in a different country. This doesn't match your typical usage pattern. Was this you?" |
| 6.7 | **Email Provenance & Trust Chain** | Visualizes SPF/DKIM/DMARC verification results as simple, understandable trust indicators. Makes email authentication accessible to non-technical users. | Automatic — displayed on every email | Header analysis + reputation scoring | Every email shows a trust indicator: Green shield (fully verified — SPF, DKIM, DMARC all pass), Yellow shield (partial verification), Red shield (verification failed or suspicious). Hovering shows details. |
