# V6: AI Ingest Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Connect Iris to a local Ollama instance for AI-powered email classification, priority scoring, and category assignment on ingest. Emails get AI metadata (intent, priority, category) automatically when synced.

**Architecture:** New `src/ai/` module with Ollama REST client and AI pipeline. Pipeline runs async after IMAP sync inserts messages. Results stored in existing nullable AI columns on messages table. Config endpoints let users configure Ollama URL and model. Frontend shows priority badges and AI categories.

**Tech Stack:** reqwest (already in Cargo.toml) for Ollama HTTP calls, existing SQLite AI columns, tokio for async pipeline.

**Scope:** Classification, priority scoring, category assignment, brief summary. Deferred: embeddings/vector store (V8), entity extraction (V6.1), dynamic category manager UI (V6.1).

### Task 2: AI Pipeline Module

Create `src/ai/pipeline.rs` — AI processing functions that call Ollama.

**Files:**
- Create: `src/ai/pipeline.rs`
- Modify: `src/ai/mod.rs`

**Implementation:**

The pipeline sends a single structured prompt to Ollama asking for JSON output with:
- intent: ACTION_REQUEST | INFORMATIONAL | TRANSACTIONAL | SOCIAL | MARKETING | NOTIFICATION
- priority: urgent | high | normal | low (with 0.0-1.0 score)
- category: one of [Primary, Updates, Social, Promotions, Finance, Travel, Newsletters]
- summary: 1-2 sentence summary

Single prompt approach (efficient — one Ollama call per email instead of 4):

```rust
pub struct AiMetadata {
    pub intent: String,
    pub priority_score: f64,
    pub priority_label: String,
    pub category: String,
    pub summary: String,
}

pub async fn process_email(client: &OllamaClient, model: &str, subject: &str, from: &str, body: &str) -> Option<AiMetadata>
```

Parse JSON response, fall back to defaults on parse failure.
