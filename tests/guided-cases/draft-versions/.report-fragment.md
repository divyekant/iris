### Key Findings

**GC-369**: `save_version` handler responds with HTTP 200 instead of HTTP 201. The route uses `Ok(Json(...))` which defaults to 200 — it should return `(StatusCode::CREATED, Json(...))`.

**GC-370**: Auto-versioning is wired into the draft update handler (`auto_version_if_changed`), so every body change creates an implicit version. Explicit POST /versions after an update yields non-consecutive version numbers (1, 3, 5 instead of 1, 2, 3). The test expected explicit saves to produce version_number 2 and 3, but they got 3 and 5 because auto-versioning consumed 2 and 4.

**GC-373**: `list_versions` does not verify the draft exists — it queries `draft_versions WHERE draft_id = ?` and returns an empty array when no rows match. This means GETs on non-existent draft IDs return 200/`[]` instead of 404. A draft-existence check (or a JOIN to messages table) is needed.

**GC-376**: Security issue — `restore_version` handler ignores the `account_id` in the request body for authorization. It fetches the version (which contains the stored `account_id`) and uses that to perform the UPDATE, bypassing any cross-account check. The handler should validate that `req.account_id` matches the version's `account_id` and return 404 if not.
