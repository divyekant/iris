## Key Implementation Details
- **Endpoints**: `POST /api/compose/scan-dlp`, `POST /api/compose/dlp-override`
- **Auth**: `X-Session-Token` required on both endpoints; missing or invalid token → 401
- **Detection types**: `credit_card`, `ssn`, `api_key`, `password`, `private_key`, `bank_account`
- **Credit card validation**: 13–19 digits, Luhn check, prefix check (Visa `4`, MC `51–55`, Amex `34/37`)
- **SSN validation**: `XXX-XX-XXXX` format, rejects all-same-digit and sequential patterns
- **API key prefixes**: `sk-*`, `pk_live_*`, `pk_test_*`, `AKIA*`, `ghp_*`, `glpat-*`, `xoxb-*`, `Bearer ey*`
- **Password pattern**: `password:`, `pwd:`, or `pass:` followed by non-whitespace
- **Private key pattern**: `-----BEGIN (RSA |EC )?PRIVATE KEY-----`
- **IBAN pattern**: 2 letters + 2 digits + 4–30 alphanumeric, minimum 15 chars total
- **Masking**: first 4 chars + `****` + last 4 chars; values ≤ 8 chars masked entirely as `****`
- **Risk levels**: `none` (no findings), `low` (passwords or single API key), `high` (credit card, SSN, private key, bank account, or multiple findings)
- **Override flow**: `findings_acknowledged: true` required to receive a one-time UUID token; `validate_override_token` removes token on first use
- **Luhn false-positive guard**: GC-367 verifies that a 16-digit Visa-prefix number failing Luhn is not reported
