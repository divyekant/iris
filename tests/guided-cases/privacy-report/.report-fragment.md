### Root Cause Summary

All failures (GC-409 through GC-417) share a single root cause: the `privacy_report.rs` handler queries `messages.has_tracking_pixels` (used in three SQL WHERE clauses and one SELECT), but **no migration has ever added this column** to the `messages` table. The column does not exist in `001_initial.sql` or any subsequent migration (001–032). Every request that reaches the DB query layer returns HTTP 500 with an empty body.

**Fix required:** Add a migration (e.g., `033_tracking_pixels.sql`) with:
```sql
ALTER TABLE messages ADD COLUMN has_tracking_pixels INTEGER NOT NULL DEFAULT 0;
```
And populate it during IMAP sync (or lazily on first access) by scanning `body_html` with the existing `extract_tracker_domains_from_html` function.

**Secondary defect (GC-413/GC-414):** `account_id` parameter is typed as `Option<String>` rather than `Option<i64>`, so non-integer values (e.g., `account_id=abc`) pass deserialization and reach the SQL layer instead of returning 400. The test cases expect 400 for non-integer `account_id`; the current implementation returns 500 (due to the DB error, which masks this separate validation gap).
