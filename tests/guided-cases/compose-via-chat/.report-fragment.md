### Notes

**Root cause for SKIP cases (GC-339, 341, 342, 346):** The AI provider in use (claude-haiku-4-5 via Anthropic OAuth) does not reliably invoke the `compose_email` tool or produce a parseable `COMPOSE_PROPOSAL:` text block in response to compose requests. In some sessions the model emits `COMPOSE_PROPOSAL:` mid-response but the `parse_action_proposal` function in `src/api/chat.rs` does not extract it — because the parser only strips text up to the marker index but the marker appears in the middle of a longer response, causing `rfind('}')` to capture extra trailing text. In other sessions the model provides a conversational draft template without any proposal marker.

**Design observation:** The COMPOSE_PROPOSAL text-based format is a fallback designed for Ollama. For Anthropic, the model is expected to use the native `compose_email` tool. However, the agentic loop in `chat.rs` does not translate a `compose_email` tool call result into a `proposed_action` on the final chat message. The tool result data is fed back into the LLM as a tool result message but the final `LlmResponse::Text` from the assistant is then passed through `parse_action_proposal` which only looks for `COMPOSE_PROPOSAL:` in text. If the model doesn't output that marker in its final summary text, `proposed_action` remains null. This is a functional gap in the compose-via-chat flow when using Anthropic native tools.

**GC-340 clarification:** Passes vacuously — since no compose proposal was produced, there was nothing to decline. The core invariant (no draft without confirm) was verified and holds.
