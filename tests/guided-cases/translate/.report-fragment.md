### Bugs Found

1. **Whitespace-only text not rejected (GC-383 step 3)** — `translate` handler checks `input.text.is_empty()` but not `input.text.trim().is_empty()`. Text of all spaces reaches the AI provider, which echoes back "No text provided to translate." Fix: add `|| input.text.trim().is_empty()` to the validation guard in `src/api/translate.rs`.

2. **`translate-email` API contract mismatch (GC-381, GC-387)** — The endpoint accepts `{subject, body, target_language, formality}` but test cases assumed a `{message_id, target_language}` interface with server-side message lookup. Either (a) update the test cases to use the actual field names, or (b) add a message-ID-based overload that fetches subject/body from the DB.

3. **AI hallucination in same-language auto-detect (GC-388 step 3)** — When `source_language` is omitted and the text is already in `target_language`, the AI sometimes ignores `target_language` and translates to English. This is an AI reliability issue, not a validation bug. Mitigation: when no `source_language` is provided, detect language first and short-circuit if it matches `target_language`.
