## Bugs Found & Fixed

### Bug 1: confirm_action SQL placeholder offset (P0 - FIXED)
- **File**: `src/api/chat.rs:271`
- **Issue**: Placeholder indexing used `i + 2` instead of `i + 1`, causing rusqlite to fail on `?3` when only 2 params bound
- **Fix**: Changed to `i + 1`
- **Impact**: All action confirmations silently failed with 500 Internal Server Error

### Bug 2: Response message ID mismatch (P0 - FIXED)
- **File**: `src/api/chat.rs:188,208`
- **Issue**: DB INSERT used one UUID (`assistant_msg_id_inner`), but API response returned a DIFFERENT UUID (`uuid::Uuid::new_v4()`) — confirm endpoint couldn't find the chat message
- **Fix**: Hoisted `assistant_msg_id` above the block scope, used same ID for both INSERT and response

### Bug 3: Truncated message ID resolution (P0 - FIXED)
- **File**: `src/api/chat.rs:256-283`
- **Issue**: AI system prompt shows emails as `[8charID]` — when producing ACTION_PROPOSAL, AI uses these 8-char prefixes as `message_ids`, but `messages` table has full UUIDs. UPDATE matched 0 rows.
- **Fix**: Added resolution step that uses `LIKE ?1||'%'` to resolve truncated IDs to full UUIDs before executing the batch UPDATE

### Bug 4: Docker Memories URL (config - FIXED)
- **File**: `docker-compose.yml:11`
- **Issue**: `MEMORIES_URL=http://memories:8900` — no Docker service named "memories" exists; the `effective_url()` function only translates `localhost`/`127.0.0.1` to `host.docker.internal`
- **Fix**: Changed to `MEMORIES_URL=http://localhost:8900`
- **Impact**: Semantic search from Memories was always failing, falling back to FTS5
