**Case-insensitive category filter (GC-030)**

- **Fix**: Changed to `LOWER(ai_category) = LOWER(?)` in both `handle_list_emails` and `handle_search_emails`.

## Key Observations

1. **Category case sensitivity was a real bug** — The tool description said lowercase but the DB stored Title Case. The AI consistently passes lowercase. Fixed with LOWER() in SQL.

2. **System prompt pre-embedding is very effective** — GC-032 showed the AI can answer detailed aggregate questions without tool calls because `chat_system_prompt_with_stats()` embeds top senders with counts. This is efficient but means inbox_stats tool is rarely called.

3. **Compound filters work perfectly** — GC-033 and GC-037 confirmed multiple filters (sender + category + is_read + sort + limit) all apply correctly in a single SQL query.

4. **Special characters handled safely** — The `%` in "70% Off" went through FTS5 MATCH (not LIKE), so no metacharacter escaping was needed. The LIKE escaping in sender filters (added in V11-S3) protects a different code path.

5. **Long queries exercise the full agentic loop** — GC-036 triggered 4 tool calls across multiple iterations, demonstrating the max_iterations=5 loop working under load. The AI intelligently decomposed the 10-question query into targeted tool calls.

6. **Search + Read chaining is reliable** — GC-034 confirmed the pattern: search to find IDs, read for full content. The AI correctly extracts the message_id from search results and passes it to read_email.
